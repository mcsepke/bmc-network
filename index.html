Rubrik 4-Node Dell Cluster — Dual Network Topology

Primary ports (Management + routable Data) uplink to the core switch · Secondary ports uplink to the isolated backup LAN

Legend

Management traffic — always flows over primary ports, egresses the core switch to Rubrik Security Cloud at bmc.my.rubrik.com on HTTPS TCP/443.
Routable data — also uses the primary ports; remains idle when every backup source and target lives on the isolated LAN. Available on demand for cloud archive, replication, or reaching hosts that aren't on the backup LAN.
Backup traffic — flows over secondary ports, contained entirely within the non-routable backup LAN. Ingest, restore, and intra-cluster replication all stay local.
Static route — added during cluster bootstrap so each Rubrik node knows to reach the protected-host CIDR via its secondary interface. No default gateway exists on the backup LAN.

Deployment notes

Port layout per node: 4 NICs total — 2 on the top edge bonded as bond0 (Primary: Management + routable Data), 2 on the bottom edge bonded as bond1 (Secondary: non-routable backup). Use LACP or active-backup bonding to match your switch configuration on both pairs.

Management is non-negotiable on primary: Rubrik Security Cloud is SaaS, so the cluster must reach bmc.my.rubrik.com on TCP/443 regardless of where backup data moves. That path lives on bond0 via the core switch's default gateway.

Static route at bootstrap: During initial configuration, add a static route on each node scoped narrowly to the protected-host subnet (e.g. 10.50.0.0/24 via bond1). Do not add a default gateway on the backup LAN — that would break the return path for management traffic.

Routable Data interface stays configured: Assigning the Data VLAN to bond0 means you can onboard a protected workload off the backup LAN, add cloud archive, or replicate to another Rubrik cluster without re-cabling or re-bootstrapping. It simply sits idle when traffic has no reason to take that path.
